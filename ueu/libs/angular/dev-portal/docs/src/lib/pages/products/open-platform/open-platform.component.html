Estrutura de customização

A estrutura de customização tem como finalidade dar a liberdade controlada para o usuário adequar o template da store de acordo com a sua proposta de serviço e conversão. A customização proporciona ao usuário alternativas de edição da estrutura da loja que inicialmente vamos definir como módulos, com uma limitação de estruturas possíveis inicialmente podemos escalar em opções de acordo com a evolução do projeto.

Módulos

Módulos são massas estruturantes que juntas formam o wireframe do template, os módulos nada mais são do que os organismos do Atomic Design sem a propriedade visual do componente afim de formatar a estrutura organizacional da página, inicialmente vamos oferecer ao usuário 4 opções de módulos sendo que apenas 3 deles podem compor uma página, essa regra existe dada a opção de módulo de valor de venda.

Módulo de Texto

O módulo de texto tem como premissa oferecer o conteúdo sobre a empresa ou produto mas também oferece liberdade textual para o usuário exibir qualquer mensagem sem limites de caracteres.

Módulo Card Banner

O módulo de Coin Value tem como premissa funcionar para vendas de valores em condição monetária, onde a marca vende um valor em R$ para consumo dentro de seu produto.

Layouts

Possibilidades de wireframe com a utilização de 2 módulos.

Blocos não modulares

Existem blocos no MVP que não são modulares ou seja eles não podem ser aplicados em um lugar diferente do já definido no wireframe, sendo assim o usuário ja inicia o desenvolvimento em um esqueleto existente, onde ele modula apenas o corpo da loja com os módulos acima mantendo uma unidade do aplicativo no topo.

Store Header

O store header é o componente que agrega a capa, avatar, back button e o fav button, ele possui personalização por parte do usuário com as imagens do avatar e capa mas não possibilita alteração de posicionamento, tamanho, cor e texto.

Fluxos de navegação

No MVP temos possibilidades de fluxo de venda que devem ser utilizados de acordo com o produto e proposta de experiência do parceiro integrado. O acesso ao fluxo de navegação pode ser feito em nosso Whimscal.

Estrutura de personalização

A estrutura de personalização tem como finalidade proporcionar ao usuário a possibilidade de personalizar os componentes da store para uma maior imersão na experiência de compra, assim como a estrutura de customização inicialmente vamos limitar o nível de personalização para alavancar métricas para validarmos a evolução dessa personalização e ainda garantirmos a unidade visual do aplicativo. A personalização garante ao usuário uma integração da sua marca com o template da store proporcionando ferramentas iniciais para criação de uma experiência integrada a marca.

Componentes personalizáveis

Header

Banner

O banner tem a possibilidade de upload de imagem de acordo com a necessidade do usuário e não possui qualquer restrição em relação ao conteúdo da imagem como excesso de texto ou elementos.

Qual o limite de texto?

Não há limite de texto para o banner da loja, mas para uma leitura mais dinâmica do leitor recomendamos o limite de 20% da imagem do banner ser composta por texto.

Avatares

O avatares tem a possibilidade de upload de imagem de acordo com a sua aplicação, eles são exibidos em diferentes conteúdos durante a navegação assim como em diferentes tamanhos, não possuem nenhuma restrição quanto ao conteúdo mas devem seguir a recomendação de serem simplificados para uma fácil leitura do usuário.

Card Button

O card button é formato por 3 atomos de texto, eles podem ser organizados de acordo com a experiência desejada, 1 atomo de ícone e 1 atomo de seta que não podem ser reposicionados dentro do card button.

Content Text

O texto que descreve a loja no usuário de modo generalista ou especifico pode ser realocado de maneira que entregue a experiência desejada (como explicado na área de módulos) ele não pode ter suas propriedades de cor, tamanho e tipo de fonte alterados.

- O Content text não possui limite de caracteres para o texto de corpo e possui o limite de linha para o título.

- Uma boa prática é utilizar textos diretos que auxiliem o usuário na tomada de decisão sobre o produto.

Green Button

O green button é utilizado para executar funções essenciais dentro de fluxos, ele é formato pelo elemento verde junto a um átomo de texto.

- O green button deve possuir 1 linha e não podem ter sua propriedade de cor e largura alterados.

- Uma boa prática é a utilização de ações dinâmicas para garantir fluidez na experiência.

Secondary Button

O secondary button é utilizado para executar funções não essenciais ou secundárias dentro de fluxos, ele é formato pelo átomo de texto.

- O secondary button deve possuir 1 linha e não pode ter sua propriedade de cor alterada.

- Uma boa prática é a utilização de ações dinâmicas para garantir fluidez na experiência.

Webhook

Quando um usuário efetuar uma compra de um produto, e essa, tiver seu pagamento(BIZ, movimentação das carteiras dentro do Picpay) efetuado com sucesso, um webhook será disparado para o parceiro de acordo com as informações fornecidas no cadastro da store.

Para a consolidação do Webhook, precisamos de 3 tipos de requests de API:

Autenticação

Request para a requisição de token de autorização a partir de login

Execução da Compra

Endpoints utilizados para fazer uma compra com sucesso
Ex: /api/compra requisição para solicitar um código e /api/confirmacao requisição para a confirmação da solicitação de compra.

Cancelamento da Compra

Endpoint utilizado para o cancelamento de compra caso ocorra um erro no fluxo

Para o Webhook de execução, podemos ter somente um endpoint.

O Webhook de autenticação só é obrigatório caso a API do parceiro exija autenticação por login.

Informações necessárias para a configuração do webhook:

Api url: (url disponibilizada pelo parceiro, a mesma deve ter certificado SSL)

Método: (método do endpoint)

HEADERS:
CHAVE DESCRIÇÃO
Content-Type application/json
HEADERS-PARCEIRO-X Headers que o parceiro poderá definir no cadastro da loja

Body

JSON especificado pelo parceiro. Por exemplo:

CHAVE DESCRIÇÃO TIPO OBRIGATÓRIO NULO OBSERVAÇÕES
transaction Informações da transação fornecidas pelo picpay Object Sim Não
transaction.identifier Identificador da transação criada no picpay String Sim Não
transaction.consumer Informações do usuário comprador do produto Object Sim Não
transaction.consumer.identifier Identificador do usuário no picpay String Sim Não
transaction.value Valor total da transação em reais Double Sim Não
transaction.created_at infoCodes de criação da transação Datetime Sim Não Podemos colocar uma regra para escolha do parceiro, para receber o valor em reais ou centavos
Informações que o parceiro poderá definir na configuração Any Sim Sim A infoCodes será enviada com timezone
Exemplo de uma request

POST /v1/compra HTTP/1.1
Host: api.parceiro.com.br
Content-Type: application/json
Request-Identifier: 0cc1f831-3ebd-4e1e-95e8-88d0df1a2830
API-KEY: KEY
API-ID: ID

Nesse exemplo de request, podemos ver que o parceiro optou por adicionar dois headers(API-KEY e API-ID) com os seguintes valores respectivamente(KEY e ID), além também, de adicionar dois valores ao body (product e username), onde essas informações ele mapeou aos componentes das telas criadas. Neste caso, vamos possibilitar que o parceiro possa mapear um componente para receber o valor preenchido ou pré definido nele, por ex: um input onde o name dele foi [ username ] e o valor foi preenchido pelo usuário, ou, um componente de seleção onde o name dele foi [ product ] e o valor pré definido foi [ 12345 ].
